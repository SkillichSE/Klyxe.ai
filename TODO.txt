KLYXE — ТЕХНИЧЕСКОЕ ЗАДАНИЕ
Продукт: RAG-чат с управлением знаниями. Цель: превзойти NotebookLM по глубине работы с источниками, гибкости моделей и контролю пользователя.

---

ЧТО СЕЙЧАС ЕСТЬ И ЧТО НУЖНО ИСПРАВИТЬ

Текущий макет реализует правильную концепцию трёхколоночного рабочего пространства - Sources, Chat, Studio и Notes - но имеет фундаментальные ограничения которые делают его непригодным для продакшена.

Весь "RAG" сейчас это просто конкатенация текста документов в контекст модели. Нет чанкинга, нет векторного поиска, нет ранжирования. При большом количестве документов это немедленно упирается в лимит токенов и качество ответов деградирует. Все данные - ключи API, сессии, документы - живут в localStorage, что означает отсутствие синхронизации между устройствами, максимум 5 МБ хранилища и потерю всего при очистке браузера. Нет авторизации, нет командной работы. Studio генерирует контент без реального учёта источников. Сравнение моделей работает поверхностно.

---

ХОСТИНГ И ДЕПЛОЙ

Фронтенд деплоится на GitHub Pages через GitHub Actions. Репозиторий содержит отдельные ветки или папки для фронтенда и конфигурации деплоя. Сборка происходит автоматически при пуше в main через workflow: установка зависимостей, сборка Vite или аналогичным бандлером, публикация папки dist на ветку gh-pages.

Поскольку GitHub Pages отдаёт статику, все динамические маршруты обрабатываются через SPA-режим: в репозитории лежит файл 404.html который редиректит на index.html с сохранением пути, либо используется хеш-роутинг. Кастомный домен подключается через файл CNAME в корне репозитория.

Все API-запросы идут на отдельный бэкенд который хостится независимо - Railway, Render, Fly.io или VPS. GitHub Pages обслуживает только статические ассеты: HTML, JS, CSS, иконки. Никаких серверных функций на уровне хостинга нет, поэтому вся логика проксирования LLM, хранения файлов и векторного поиска живёт исключительно на бэкенде.

Переменные окружения на фронтенде ограничены публичными значениями: URL бэкенда, публичный Sentry DSN если используется мониторинг ошибок. Никаких секретов в репозитории и в собранном бандле.

АРХИТЕКТУРА БЭКЕНДА

Стек: Node.js или Python/FastAPI, PostgreSQL, pgvector или Qdrant для векторного хранилища, S3/R2/MinIO для файлов, BullMQ или pg-based queue для фоновых задач.

Аутентификация реализуется через email и magic link или OAuth через Google. Паролей на первом этапе нет. Сессии через JWT в httpOnly cookies с refresh token rotation.

API-ключи провайдеров пользователь вводит свои - OpenRouter, OpenAI, Anthropic, Gemini. Ключ шифруется на бэкенде через AES-256 и хранится в БД привязанным к аккаунту. Фронт никогда не держит ключ в сыром виде. Все запросы к LLM идут через бэкенд-прокси.

Основные модели данных: User (id, email, created_at), Notebook (id, user_id, title, created_at, updated_at), Source (id, notebook_id, type, name, status, raw_text_path), Chunk (id, source_id, text, embedding, token_count, chunk_index), ChatSession (id, notebook_id, title, created_at), Message (id, session_id, role, content, citations как jsonb), Note (id, notebook_id, content, created_at, updated_at), StudioOutput (id, notebook_id, tool, content, created_at).

PIPELINE ЗАГРУЗКИ ИСТОЧНИКОВ

Это самое важное отличие от NotebookLM. Когда пользователь добавляет источник, процесс выглядит так.

Файл загружается через multipart upload на облачное хранилище, статус источника сразу становится "indexing". Задача уходит в очередь и воркер начинает обработку: PDF извлекается через pdf-parse или pdfplumber, сканированные PDF прогоняются через OCR (Tesseract или Mistral OCR API), URL-источники загружаются через Playwright для JS-рендеринга и очищаются через readability, YouTube-ссылки транскрибируются через yt-dlp или YouTube Data API, аудиофайлы через Whisper API.

Извлечённый текст разбивается на чанки по 400-600 токенов с overlap 100 токенов. Чанки стараются быть семантически осмысленными - сначала разбивка по параграфам, резка по токенам только если параграф слишком большой. Для каждого чанка батчом генерируются эмбеддинги через text-embedding-3-small или large, по 100 чанков за раз. Эмбеддинги сохраняются в pgvector.

Когда обработка завершена, источник переходит в статус "ready" и фронт получает обновление через WebSocket или Server-Sent Events без перезагрузки страницы. Если произошла ошибка - статус "error" с понятной причиной и кнопкой Retry.

Отдельная оптимизация: если тот же текст уже был проиндексирован (по хешу содержимого) - эмбеддинги переиспользуются, повторная генерация не запускается.

RAG-PIPELINE ДЛЯ ЧАТА

Когда пользователь отправляет сообщение, бэкенд генерирует query embedding для этого сообщения и выполняет векторный поиск по top-8 до top-12 чанков из выбранных источников по косинусному расстоянию. Если пользователь не выбрал конкретные источники - поиск идёт по всему ноутбуку.

Опционально найденные чанки прогоняются через reranker (Cohere Rerank или bge-reranker) и список сужается до top-5 наиболее релевантных. Чанки вставляются в системный промпт с номерами и инструкцией модели ссылаться на источники через [1], [2] при цитировании.

Ответ стримится на фронт через SSE. По окончании стрима парсятся цитаты из ответа и сохраняются в Message.citations как массив объектов с chunk_id, source_id, именем источника и цитируемым фрагментом. Под каждым ответом в интерфейсе отображаются кликабельные чипы с названием источника и номером страницы. Клик открывает правую панель с превью документа, прокрученным к нужному фрагменту. Именно это NotebookLM не умеет - у них нет превью источника.

---

УПРАВЛЕНИЕ ИСТОЧНИКАМИ

Список источников сохраняет логику из макета с чекбоксами для выбора активных источников. Дополнительно показывается сводка: сколько токенов и чанков задействовано из выбранных источников.

Для каждого источника доступна кнопка Preview которая открывает правую панель с рендером: PDF через PDF.js, URL через iframe с sandbox, текст через форматированный вид.

Пользователь может задавать теги источникам (paper, book, web, data и т.д.) и фильтровать список по ним. Доступны bulk-операции: выбрать несколько, удалить, добавить тег, переместить в другой ноутбук.

URL-источники поддерживают опциональный auto-refresh по расписанию - раз в день или в неделю. Если содержимое изменилось - источник переиндексируется и пользователь получает уведомление. Для любого источника доступна кнопка "что изменилось с прошлого раза" если он обновлялся.

---

МУЛЬТИМОДАЛЬНОСТЬ

Изображения в PDF извлекаются и описываются через vision-модель (GPT-4o или Claude). Описания добавляются как отдельные чанки с префиксом [Image description:]. Таблицы из PDF обрабатываются через camelot или pdfplumber и сохраняются в markdown-формате как отдельные чанки. Это позволяет задавать вопросы к табличным данным.

STUDIO

Studio должен стать настоящим инструментом, а не просто промпт-формой. Все инструменты используют реальный RAG-pipeline, не просто берут весь текст источников.

Summaries генерирует резюме по каждому источнику отдельно и общее резюме ноутбука. Результаты кэшируются и отображаются рядом с источником.

Key Claims извлекает конкретные утверждения с подтверждающими цитатами.

Compare Sources при выборе двух и более источников строит структурированное сравнение позиций.

Timeline пытается выстроить хронологию событий из документов.

Draft генерирует черновик - эссе, отчёт, письмо - на основе источников. Пользователь задаёт тип, длину, аудиторию и тон. После генерации доступен редактор с возможностью редактировать отдельные параграфы и перегенерировать их.

Mindmap генерирует JSON-структуру концептов и рендерит её визуально через D3 или ECharts прямо в панели Studio.

Все выходы Studio сохраняются в StudioOutput и доступны в истории.

---

NOTES

Textarea заменяется блочным редактором на базе Tiptap или BlockNote с поддержкой заголовков, списков, таблиц, блоков кода и markdown-форматирования.

Поддерживается inline-цитирование: пользователь вставляет цитату из источника через drag из чата или команду /cite. Вставляется блок с текстом цитаты и ссылкой на источник. Автосохранение происходит каждые 5 секунд с debounce.

Экспорт доступен в Markdown, PDF (через headless Puppeteer на бэкенде) и DOCX. Можно сгенерировать публичную read-only ссылку на заметку.

СРАВНЕНИЕ МОДЕЛЕЙ

Два параллельных запроса превращаются в полноценную фичу. Сохраняются результаты с метаданными: модели, время ответа, количество токенов, приблизительная стоимость. Пользователь может отметить лучший ответ - это накапливается в личной аналитике. Кнопка Diff реально показывает визуальный diff между ответами с выделением уникальных фрагментов. Отдельный раздел хранит историю всех прошлых сравнений.

НОУТБУКИ И ОРГАНИЗАЦИЯ

При входе пользователь попадает на дашборд со списком ноутбуков - карточки с названием, датой, количеством источников и последней активностью. Доступны создание, переименование, дублирование, удаление и архивирование.

Шаблоны ноутбуков: Research Paper, Competitive Analysis, Book Notes, Legal Review - с предзаполненными системными промптами и наборами инструментов Studio.

Глобальный поиск охватывает названия источников, содержимое чата и заметки через семантический поиск по эмбеддингам. Пользователь может написать "что я спрашивал про методологию" и найти конкретные сообщения из прошлых сессий.

Внутри каждого ноутбука доступна небольшая аналитика: сколько источников проиндексировано и сколько токенов они занимают, какие источники чаще всего цитировались, самые популярные темы из истории чата, когда ноутбук открывался последний раз.

---

ПЕРСОНАЛИЗАЦИЯ

В настройках пользователь заполняет поле "Как я работаю" - например "Я PhD в молекулярной биологии, отвечай технически" или "Я маркетолог, упрощай язык и давай практические выводы". Этот промпт автоматически вставляется во все запросы.

После каждого ответа в чате отображаются не статичные suggestion cards, а динамически сгенерированные follow-up вопросы на основе содержания только что полученного ответа. Один дешёвый запрос к модели - и пользователь продолжает исследование не задумываясь что спросить дальше.

---

ОНБОРДИНГ

При первом входе интерактивный онбординг из трёх шагов прямо в интерфейсе: загрузи источник, задай вопрос, сохрани в заметки. Не тур с подсказками, а живой демо-ноутбук который уже заполнен. Пользователь видит как работает продукт до того как загрузит свой первый файл.

---

PWA И OFFLINE

Service Worker позволяет работать с ноутбуком без интернета в режиме чтения: просматривать источники, читать историю чата и заметки. При восстановлении соединения - синхронизация. Актуально для исследователей в поездках или с нестабильным интернетом.

GitHub Pages поддерживает PWA нативно: манифест и Service Worker отдаются как статические файлы, HTTPS включён по умолчанию. Установка на рабочий стол и мобильный работает без дополнительной конфигурации.

---

ПУБЛИЧНЫЙ API

REST API для разработчиков: создание ноутбуков, добавление источников, отправка запросов к RAG-pipeline программно. Webhook-уведомления об окончании индексации. NotebookLM API существует но крайне ограничен. Klyxe с открытым API открывает другую аудиторию: разработчики, автоматизации, интеграции с n8n и Zapier.

---

БЕЗОПАСНОСТЬ

API-ключи пользователей шифруются в БД и никогда не логируются и не передаются на фронт. Rate limiting по IP и по user_id - лимиты на загрузку файлов, запросы к LLM и создание эмбеддингов. Валидация файлов по MIME-type и magic bytes. Политика CORS на бэкенде ограничена доменом GitHub Pages продукта. CSP-заголовки запрещают inline-скрипты и внешние скрипты кроме разрешённых - настраиваются через мета-теги в HTML так как GitHub Pages не позволяет управлять серверными заголовками. Все запросы к LLM только через бэкенд-прокси.

ПРИОРИТЕТ РАЗРАБОТКИ

Этап 1, около 6 недель: авторизация, хранение файлов на S3, реальный RAG с pgvector, проксирование LLM через бэкенд, базовый чат с цитатами, список ноутбуков, онбординг, настройка деплоя на GitHub Pages через GitHub Actions.

Этап 2, около 4 недель: Studio с реальным RAG, Notes с Tiptap, превью документов в правой панели, экспорт заметок, follow-up вопросы.

Этап 3, около 3 недель: сравнение моделей с историей, Mindmap, семантический поиск по чату, auto-refresh источников, аналитика ноутбука.

Этап 4, около 2 недель: публичный API, PWA.

ГЛАВНОЕ КОНКУРЕНТНОЕ ОТЛИЧИЕ ОТ NOTEBOOKLM

NotebookLM привязан к Google-моделям, не показывает превью источников по клику, не позволяет выбирать какие источники активны для запроса, не даёт сравнивать модели, не поддерживает собственные API-ключи, имеет слабый редактор заметок, не поддерживает командную работу нормально и не обновляет источники автоматически. Klyxe закрывает все эти пробелы и делает ставку на model-agnostic подход, глубокую работу с источниками и открытый API.
